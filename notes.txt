The static sandbox lets us verify that our application can successfully 
call the Selling Partner API using the full authentication flow. 
This includes:

- LWA tokens, which identify our application in Developer Central 
  and authorize access on behalf of the seller

- SigV4 signing, which is required for every SP‑API request to ensure 
  the request is secure and hasn’t been tampered with

- AWS credentials, which SigV4 uses to cryptographically sign each request 
  through Amazon’s API Gateway. The AWS account is linked to the SP-API 
  developer profile, so only the person with that IAM role and those AWS 
  credentials will be able to call the endpoints. For static sandbox, this 
  linking isn't available so any aws account will work by providing 
  credentials.

- Per request signing, meaning every API call must be individually signed 
  using SigV4 and our AWS keys


While the static sandbox is useful for confirming that authentication and 
request signing are working correctly, it contains very limited and 
unrealistic financial data. Because of this, it isn’t suitable for building 
or testing real application logic. To continue development, we generated 
mock financial event data that accurately reflects real SP‑API responses, 
allowing us to simulate real‑world behavior and build the profit‑calculation
 features of the application.
